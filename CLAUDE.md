# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Overview

Unity 6 game project, codenamed **Bladehold**. A 3D action game built on Unity's Universal Render Pipeline (URP) with the new Input System and NavMesh-based AI. The player fights waves of melee goblins, earns gold, and dies/restarts. Unity Editor version is pinned in `ProjectSettings/ProjectVersion.txt` (currently `6000.3.10f1`) — open with exactly this version.

## Building, running, and testing

There is no command-line build script. Day-to-day work happens in the Unity Editor:

- **Open the project**: open the project folder in Unity Hub with the matching Editor version.
- **Play / run**: enter Play mode in the Editor. The only scene in the build list is `Assets/Scenes/SampleScene.unity` (see `ProjectSettings/EditorBuildSettings.asset`).
- **Build a player**: File > Build Profiles / Build Settings. URP renderers are pre-configured for two targets — `Assets/Settings/PC_Renderer.asset` and `Assets/Settings/Mobile_Renderer.asset`.
- **Tests**: `com.unity.test-framework` is installed. Run via the Editor's **Test Runner** window (Window > General > Test Runner). There are no test assemblies in `Assets/` yet — a new one needs its own `.asmdef` to be picked up.

The `.csproj` and `.slnx` files are **generated by Unity** for IDE integration (Rider / Visual Studio). Do not hand-edit them; edit code under `Assets/` and let Unity regenerate. C# compiles automatically on focus/save in the Editor.

## Source control

**Commit and push directly to `main`.** This is a solo project — do not create feature branches or open PRs unless explicitly asked. When the user says "commit and push" (or similar), commit straight onto `main` and push to `origin/main`.

## Code layout

First-party game code lives **only** in `Assets/Bladehold/Bladehold Scripts/` (note the space in the folder name), organized into subfolders: `Player/`, `Enemies/`, `DamageSystem/`, `Economy/`, `Save/`, `UI/`, `Waves/`, `Stats/`, `Upgrades/`, `Reincarnate/`, plus `Editor/` for editor-only tooling (compiled out of builds by Unity's special-folder rule). Designer-editable data that isn't a `ScriptableObject` lives in `Assets/Bladehold/Config/` (`SkillTree.csv`, `Reincarnate.csv`, `Enemies.csv`).

Everything else under `Assets/` is vendored and should generally **not** be modified:
- `Assets/Third Party/` — Synty animations (incl. the **active player movement controller**, see below), Unity StarterAssets (present but **not** the controller in use), Wingman (editor tool), Kevin Iglesias animations.
- `Assets/LeanTween/`, `Assets/DamageNumbersPro/`, `Assets/Feel/` (MoreMountains Feel/Feedbacks — `MMF_Player`), `Assets/AssetInventory/` — vendored asset-store packages used by game code.

There are **no `.asmdef` files for game code**, so all first-party scripts compile into the default `Assembly-CSharp`. The only first-party-adjacent assembly definition is `Assets/Third Party/Wingman/Wingman.asmdef` (an editor productivity tool, not game logic).

## Architecture

Gameplay centers on the player fighting waves of melee goblins. The design leans heavily on **`ScriptableObject` config**, **event-driven decoupling**, and a few **scene singletons**. The most important idea to internalize is the `Health` event model (see Conventions) — almost every system hangs off it.

### Player (`Player/`)
> **Movement controller:** the player uses the vendored **`Synty.AnimationBaseLocomotion.Samples.SamplePlayerAnimationController`** (`Assets/Third Party/Synty/AnimationBaseLocomotion/Samples/Scripts/`), **not** Unity StarterAssets' `ThirdPersonController`. It drives a `CharacterController`, reads input through Synty's **`InputReader`** (which exposes `onAttackActivated`/`onAttackDeactivated` press/release events plus aim/sprint/crouch/etc.), and derives per-gait movement from the **private** `_walkSpeed`/`_runSpeed`/`_sprintSpeed` fields. Attack animation is driven by the `StartAttack` trigger + `IsHoldingAttack` bool. Neither Synty nor StarterAssets has an `.asmdef`, so these types are reachable from `Assembly-CSharp` (first-party code references them directly; `PlayerMoveSpeedBinder` reaches the private speed fields by reflection rather than editing vendored source).

- **`Player`** (`Player.cs`) — singleton (`Player.Instance`, set in `Awake`). Caches and exposes the player's `Health`, `IDamageable`, `Wallet`, and **`PlayerStats`**. Everything locates the player through this static reference rather than scene lookups.
- **`PlayerDeath`** (`PlayerDeath.cs`) — on `Health.OnDied`, disables an inspector-assigned list of control components (movement/input, kept decoupled from the vendored controller by type) and fires a `Death` animator trigger. Does **not** run if `DeathNova` cancels the death first (see below) — `OnDied` never fires for that hit.
- **`AnimationEvents`** (`AnimationEvents.cs`) — receiver for Unity animation events on the player rig; e.g. `OneHandedSwordAttack()` activates the sword's `DamageTrigger` at the right animation frame, and `PlaySwordWoosh()` (called earlier in the swing) plays the swing sound via `SwordHitFeedback`.
- **`SwordChargeFeedback`** (`SwordChargeFeedback.cs`) — escalating charge-up feedback. Polls `PlayerAttack.IsCharging`/`ChargeLevel` (same style `PlayerMoveSpeedBinder` polls stats) and plays the next `MMF_Player` in an inspector-assigned array each time the hold gains another charge level (stage N plays at level N+1; levels beyond the array keep the last stage's look).
- **`VampiricBlade`** (`VampiricBlade.cs`) — gold-tree skill line. Listens to the sword `DamageTrigger.OnHit` (the `SwordHitFeedback` pattern) and heals the player `StatType.LifeStealPercent` × damage dealt via `Health.Heal`. The sword trigger must be assigned explicitly (no auto-wire — the player has other `DamageTrigger`s like the nova hitbox). Base 0 = locked.
- **`DamageBlocker`** (`DamageBlocker.cs`) — the "Solid" gold-tree skill line. Hooks `Health.TryBlockDamage` on the player's `Health` and negates one incoming hit every `StatType.BlockCooldown` seconds (base 0 = locked; the first node sets 10s, later tiers subtract). Optional block `MMF_Player` feedback.
- **`AttackCancelsSprint`** (`AttackCancelsSprint.cs`) — pressing attack always cancels sprinting (independent of any skill-tree state; the "Sprinter" nodes just raise `StatType.SprintSpeed`). Subscribes to `InputReader.onAttackActivated` and invokes the controller's private `DeactivateSprint()` by cached reflection (the `PlayerMoveSpeedBinder` precedent) — the method rather than the `_isSprinting` field, since it also restores strafe state. Sprint stays off until the player presses sprint again.
- **`DeathNova`** (`DeathNova.cs`) / **`DeathNovaSO`** — Reincarnate-tree ability. Hooks `Health.TryPreventDeath` on the player's own `Health`; if unlocked (`StatType.DeathNovaCharges >= 1`) and off cooldown, it activates a `DamageTrigger` child hitbox (damage + knockback on nearby enemies) on every lethal hit, then — only if `StatType.DeathNovaRevivePercent > 0` — calls `Health.Revive` and cancels the death. The base node is blast-only; a separate stacking Reincarnate node adds the revive.
- **`GoldOnDeathCollector`** (`GoldOnDeathCollector.cs`) — Reincarnate-tree ability ("Grave Robber"). On the player's real `Health.OnDied` (i.e. death that `DeathNova` didn't cancel), auto-collects `StatType.GoldOnDeathPickupPercent` of the gold currently on the ground into the `Wallet`.

### Damage system (`DamageSystem/`) — the combat core
- **`IDamageable` / `Damage` / `DamageType`** (`Damageable.cs`) — the damage interface (`ReceiveDamage(Damage)`), the damage payload (`value`, `type`), and the type enum (`sharp`/`blunt`/`elemental`).
- **`Health`** (`Health.cs`) — implements `IDamageable`. Holds `currentHealth` initialized from a `HealthSO`. Raises **`OnDied`** once when health first hits zero (latches via `IsDead`; further damage ignored) and **`OnDamaged(Damage)`** on every hit. Plays an `MMF_Player` damage feedback. Before latching, it checks **`TryPreventDeath`** (an `event Func<bool>`) — a handler that returns `true` (e.g. `DeathNova` revivifying the player via `Revive(amount)`) cancels the death outright, so `OnDied` never fires for that hit. At the very top of `ReceiveDamage` it similarly checks **`TryBlockDamage`** (an `event Func<Damage, bool>`) — a `true` handler (e.g. `DamageBlocker`'s "Solid" auto-block) negates the hit entirely: no health loss, no feedback, no events. Also exposes **`Heal(amount)`** (clamped to `MaxHealth`, ignored while dead — used by `VampiricBlade` lifesteal). Health knows nothing about what reacts to these events.
- **`HealthSO`** — `maxHealth`. **`DamageSO`** — `baseDamage`, `isCritical`. **`DamageTriggerSO`** — `radius`, `duration`, `maxHits`.
- **`DamageTrigger`** (`DamageTrigger.cs`) — an activatable hitbox with two `detectionMode`s. `Activate()` (called from an animation event) opens it for `duration`; each `FixedUpdate` while active it detects hits, applies `DamageSO`/stat-driven damage, dedupes targets with a `HashSet`, and **never damages its owner** (the nearest `IDamageable` up the parent hierarchy, or an explicit `owner`). **`Sphere`** mode (the default, used by e.g. the Death Nova blast) `OverlapSphere`s `damageTriggerSO.radius` around the trigger's own position — unchanged from before. **`BladeSweep`** mode (the sword) samples `basePointCount` points along `bladeBase`→`bladeTip` and each tick raycasts every point from its *previous* position to its *current* one, so a fast/long swing can't tunnel past a thin collider between frames. In both modes, the effective cut-through cap is `StatType.MaxHitsPerSwing` when `readsPlayerStats` is on (else `damageTriggerSO.maxHits`); reaching the cap no longer ends the activation early — it keeps sweeping for the full `duration`, and only the *next new* target beyond the cap cancels it: no damage is applied to that target, `OnBlocked` fires, and the activation ends immediately. Every other successful hit raises `OnHit(IDamageable, Damage, Vector3 hitPoint)`. When `readsPlayerStats` is on, `StatType.SwordRange` is a unitless multiplier (base 1.0, same convention as `MoveSpeed`) that both scales `basePointCount` for that swing and — via an `OnStatChanged` subscription — the sword's own `transform.localScale`, so upgrading range visibly lengthens the blade.
- **`SwordHitFeedback`** (`SwordHitFeedback.cs`) — reacts to the sword's `DamageTrigger.OnHit`/`OnBlocked` the same way `DamageNumberSpawner` reacts to `Health.OnDamaged`: plays a cutting sound (a distinct pool on crits) and spawns a blood-particle burst at the hit point, sized and sped up with damage up to a cap; on `OnBlocked`, fires a `Blocked` animator trigger. Also exposes `PlayWoosh()` for the swing's animation event.
- **`DamageNumberSpawner`** (`DamageNumberSpawner.cs`) — listens to `Health.OnDamaged`, spawns a DamageNumbersPro popup that follows the target; uses an alternate `critPopupPrefab` instead of the normal one when the hit was critical, if one is assigned.
- **`DisableCollidersOnDeath`** (`DisableCollidersOnDeath.cs`) — listens to `Health.OnDied`, disables all child colliders so a corpse stops blocking/absorbing hits.

### Enemies (`Enemies/`)
- **`AIMovementSO`** — pathfinding (`speed`, `updateInterval`), gait speeds (`walkSpeed`/`runSpeed`/`sprintSpeed`, used to derive blend thresholds), optional `leanCurve`.
- **`AIMovement`** (`AIMovement.cs`) — drives a `NavMeshAgent` to chase `Player.Instance`, re-issuing `SetDestination` only every `updateInterval` seconds. Stops on its own `Health.OnDied` and on the player's death.
- **`LocomotionAnimator`** (`LocomotionAnimator.cs`) — plain (non-`MonoBehaviour`) reusable class extracted from Synty's `SamplePlayerAnimationController`. Computes locomotion blend params (`MoveSpeed`, `CurrentGait`, strafe/shuffle, `LeanValue`, …) from a per-frame `LocomotionInput` and writes them to an `Animator`. Input-agnostic; never moves the transform.
- **`AIAnimation`** (`AIAnimation.cs`) — maps a goblin's `NavMeshAgent` motion onto `LocomotionAnimator.LocomotionInput` each frame. Fires a `Death` trigger on death and a `Cheer` trigger when the player dies.
- **`AIAttackSO`** / **`AIAttack`** (`AIAttack.cs`) — melee attack. When the player is within `attackRange` and off cooldown, fires an `Attack` animator trigger; after `windupToApex` seconds, if the player is still alive and in range, applies damage. Never touches the `NavMeshAgent`, so goblins attack while chasing.
- **`EnemySO`** / **`CoinDropper`** (`CoinDropper.cs`) — on `Health.OnDied`, drops a `Coin` pickup worth a rolled amount from `EnemySO` (or a per-instance `SetCoinDrop` override from the enemy roster CSV).
- **`EnemyRosterSO` / `EnemyDefinition`** (`EnemyRosterSO.cs`) — the enemy-type roster, parsed from `Config/Enemies.csv` (same designer-edits-a-spreadsheet pattern as `SkillTreeSO`; both share `CsvUtil.SplitLine`). Columns: `id,displayName,health,damage,minGold,maxGold,speed,scale,unlockWave,spawnChance,minSpawn,maxConcurrent`. The stat columns are **optional overrides** — blank keeps the prefab's own SO value; the shared SOs are never mutated. The **first row is the unlimited fallback type** (goblin); later rows carry `unlockWave` (first wave they can appear), `spawnChance` (per-spawn roll, authored as a **percent**: 10 = 10%), `minSpawn` (guaranteed spawns of the type per wave once unlocked, filled before any chance rolls), and `maxConcurrent` (max alive at once, blank/0 = unlimited). `Health`/`AIAttack`/`CoinDropper`/`AIMovement` each expose a per-instance setter (`SetMaxHealth`/`SetDamage`/`SetCoinDrop`/`SetSpeed`) that `WaveSpawner` calls right after `Instantiate`, before the instance's `Start` (the `MarkGolden` timing trick).
- **`Enemy`** (`Enemy.cs`) — marks a GameObject as an enemy; on `Health.OnDied` reports the kill to `GameStats`.
- **`GoldenGoblin`** (`GoldenGoblin.cs`) — Reincarnate-tree ability. `WaveSpawner` rolls `StatType.GoldenGoblinChance` per spawn and calls `MarkGolden()` right after `Instantiate` (before `Start` runs). A golden goblin swaps to a gold material and, independently of `CoinDropper`, drops its own bonus `Coin` (worth `StatType.GoldenGoblinGoldBonusPercent` of a fresh `EnemySO` roll) plus optional VFX/SFX on `Health.OnDied`.

### Economy (`Economy/`)
- **`GameStats`** (`GameStats.cs`) — per-run scoreboard **singleton** (`GoblinsKilled`, `GoldEarnedThisRun`). Lives in the scene, so it resets on scene reload. Raises `OnChanged`.
- **`Wallet`** (`Wallet.cs`) — the player's coin purse. Loads its total from disk (`SaveSystem`) on `Awake` and saves on every change, so gold **persists across runs**. Raises `OnCoinsChanged`.
- **`Coin`** (`Coin.cs`) — a pickup; on trigger-enter by the `Wallet` holder it deposits to the `Wallet` and `GameStats`, shows a popup, and destroys itself.

### Save (`Save/`)
- **`SaveData`** (`SaveData.cs`) — `[Serializable]` snapshot of persisted progress: `totalGold`, `purchasedNodeIds` (gold tree, cleared on Reincarnate), `reincarnatePoints` and `purchasedReincarnateNodeIds` (Reincarnate tree — never cleared). Add fields here; old saves load missing fields as defaults.
- **`SaveSystem`** (`SaveSystem.cs`) — static JSON read/write under `Application.persistentDataPath`. Load is resilient (missing/corrupt file → fresh `SaveData`).

### Waves (`Waves/`)
- **`WaveConfigSO`** — wave-size and pacing tunables: `baseGoblinCount`, `goblinsAddedPerWave`, `maxConcurrent`, `timeBetweenWaves`, `spawnInterval`; `GoblinsForWave(n)` returns the total for wave `n`.
- **`WaveSpawner`** (`WaveSpawner.cs`) — **singleton** (`WaveSpawner.Instance`). Runs an endless countdown → spawn → clear → next-wave loop. Caps concurrent enemies at `maxConcurrent` and trickles in replacements (one per `spawnInterval`) as enemies die, until the wave's total has been killed. **What spawns comes from an `EnemyRosterSO`** plus an inspector list mapping each CSV `id` to a prefab (rows without a prefab mapping are skipped with a warning): each spawn slot checks the non-fallback rows in CSV order and spawns the first that is unlocked, under its own per-type concurrent cap, and wins its `spawnChance` roll — else the fallback first row (goblin). Roster stat overrides are applied to the instance right after `Instantiate` (see `EnemyRosterSO` above). Snaps spawn positions to the NavMesh. Tracks each enemy's death via its `Health.OnDied` (enemies are corpses, not destroyed — see Conventions), decrementing the per-type live count. Stops on player death. Exposes `CountdownTick`/`WaveStarted`/`WaveCleared` events and `CurrentWave`.
- **`WaveUI`** (`WaveUI.cs`) — listens to the spawner and drives a TMP canvas: "Wave starts in {n}", "BEGIN", "Wave Cleared!", plus an optional persistent "Wave {n}" label.
- **`RunState`** (`RunState.cs`) — static `StartingWave`, the one piece of run state that must **survive a scene reload within a session** (statics persist across `LoadScene` but reset when play stops). The spawner keeps it at the current wave; the death screen sets it to 1 or the current wave to choose where a restart resumes.

### UI (`UI/`)
- **`CoinUI`** (`CoinUI.cs`) — binds a TMP label to the player's `Wallet.OnCoinsChanged`.
- **`DeathScreen`** (`DeathScreen.cs`) — fades in on player death (via `Player.Instance.Health.OnDied`), shows run stats and total gold, and offers two restarts: from wave 1, or from the wave the player died on (the optional current-wave button reads `WaveSpawner.Instance.CurrentWave` and sets `RunState`). Both reload the active scene. An optional third button calls `ReincarnateService.Instance.Reincarnate()` instead (see Reincarnate below), previewing the points via `PreviewPointsForReincarnate()`.

Navigation depends on a baked NavMesh in the scene (`com.unity.ai.navigation`); AI agents (and wave spawning onto the NavMesh) will not work without it.

### Stats & upgrades (`Stats/`, `Upgrades/`)
A CSV-driven skill tree, shown on the death screen, where gold buys branching, stackable upgrades. Built on a central stat layer so combat/movement code reads *effective* values without the underlying configs ever being mutated.
- **`StatType` / `ModifierKind`** (`Stats/StatType.cs`) — the shared vocabulary of modifiable stats (`SwordDamage`, `SwordRange`, `MaxHitsPerSwing`, `GoldDropMultiplier`, `MoveSpeed`, `SprintSpeed`, `CritChance`, `CritMultiplier`, `KnockbackForce`, `ChargeDamageBonus`, `ChargeKnockbackBonus`, `MaxChargeLevels`, `LifeStealPercent`, `BlockCooldown`, plus the Reincarnate-only `DeathNovaCharges`/`DeathNovaCooldown`/`DeathNovaRevivePercent`/`GoldenGoblinChance`/`GoldenGoblinGoldBonusPercent`/`GoldOnDeathPickupPercent`) and modifier kinds (`Flat`, `Percent`).
- **`PlayerStats`** (`Stats/PlayerStats.cs`) — the aggregation layer, reached via `Player.Instance.Stats`. Each stat is `final = (base + Σflat) × (1 + Σpercent)`. **Bases are registered at runtime** by whichever system owns the value (so the SOs/controller stay the source of base truth, never mutated); upgrades call `AddModifier`; consumers call `GetValue`. Raises `OnStatChanged`. Modifiers can be added before or after a base is set — order-independent. `MoveSpeed`/`SprintSpeed`/`SwordRange`/`GoldDropMultiplier` are unitless **multiplier** stats (base 1.0; `GoldDropMultiplier`'s base is registered by `WaveSpawner` next to the Golden Goblin stats and consumed by `CoinDropper`).
- **`SkillNode` / `SkillEffect` / `SkillTreeSO`** (`Upgrades/`) — `SkillTreeSO` (`Scriptable Objects/SkillTreeSO`) parses a designer-edited CSV `TextAsset` (columns `id,displayName,description,cost,stat,kind,amount,prereqs,x,y,icon`) into `SkillNode`s, currency-agnostic (`cost` is just an int — both the gold tree's `Config/SkillTree.csv` and the Reincarnate tree's `Config/Reincarnate.csv` use a `SkillTreeSO` asset each). Blank `stat` = connector/unlock-only node (`effects` empty). The trailing `icon` column is optional (absent column or blank cell = no icon) and names a sprite in the `SkillTreeSO` asset's serialized `icons` list — `SkillTreeSO.GetIcon(name)` resolves it, and `SkillNodeView` shows it in an optional `icon` Image (hidden when unresolved), so each tree asset maps CSV icon names to sprites in the inspector. `stat`/`kind`/`amount` may each be `;`-separated lists of equal length to apply several `SkillEffect`s atomically from one purchase (e.g. Golden Goblin tiers bump both a chance stat and a bonus-% stat together). `prereqs` is `;`-separated (blank = a root, visible from start; otherwise revealed once any prereq is bought). Duplicate effects across different ids are intentional — buying both **stacks**.
- **`ISkillTreeService`** (`Upgrades/ISkillTreeService.cs`) — the shared contract (`Tree`, `IsPurchased`, `IsRevealed`, `CanPurchase`, `TryPurchase`, `OnTreeChanged`) both `SkillTreeService` and `ReincarnateService` implement, so `SkillTreeView`/`SkillNodeView` can render either tree without knowing which currency it spends.
- **`SkillTreeService`** (`Upgrades/SkillTreeService.cs`) — scene **singleton** (`Instance`). Owns purchase logic (`TryPurchase` spends gold via `Wallet.TrySpend`, applies the node's `effects` to `PlayerStats`), reveal/affordability queries, and **persistence**: purchased node ids live in `SaveData.purchasedNodeIds` and are **re-applied on every run** in `Start`, making upgrades permanent meta-progression like gold — until a Reincarnate wipes them (see below). Raises `OnTreeChanged`.
- **`SkillTreeView` / `SkillNodeView`** (`UI/`) — render a tree (driven by any `ISkillTreeService`) on a canvas: lay out nodes at their CSV coords, draw prereq connectors, tint by state (hidden/available/owned), buy on click, refresh on `OnTreeChanged`/`Wallet.OnCoinsChanged`. `SkillNodeView.costSuffix` labels the currency (blank for gold, e.g. `" pts"` for Reincarnate).
- **`SkillTreeCsvEditorWindow`** (`Editor/SkillTreeCsvEditorWindow.cs`) — editor window (**Bladehold > Skill Tree Editor**) for editing a `SkillTreeSO`'s CSV in-Editor: list + detail node editing, add/duplicate/delete rows, and drag-and-drop Sprite icon assignment (dropping a sprite sets the node's icon name **and** adds it to the SO's `icons` list). "Save CSV" writes the file back through `AssetDatabase` and calls `tree.Reload()`, so parse errors surface immediately.

How stats reach gameplay: **`DamageTrigger`** (with `readsPlayerStats` on) pulls effective sword damage/range/cut-through-cap from `PlayerStats`, rolls crits, applies the charge multiplier (and scales knockback by `1 + ChargeLevel × ChargeKnockbackBonus` — the "Amplified Knockback" line), and stamps `knockbackForce`/`sourcePosition` onto the `Damage`; with `readsPlayerStats` off (e.g. the Death Nova hitbox) it instead uses its own serialized `knockbackForce`. **`KnockbackReceiver`** (on goblins) reacts to `Health.OnDamaged` to shove the `NavMeshAgent`. **`PlayerAttack`** times the Synty `InputReader` attack hold in discrete charge levels — each level takes another `chargeTimePerLevel` second of holding, capped at `StatType.MaxChargeLevels` (base 0 = hold-to-charge locked until the "Heavy Strike" node is bought), and `AttackDamageMultiplier = 1 + ChargeLevel × ChargeDamageBonus` — exposing a live `AttackDamageMultiplier`/`IsCharging`/`ChargeLevel`, which **`SwordChargeFeedback`** polls to escalate charge VFX/SFX. **`PlayerMoveSpeedBinder`** scales the Synty controller's private speed fields by reflection. `SaveSystem` caches a single shared `SaveData` so `Wallet`, `SkillTreeService`, and `ReincarnateService` don't clobber each other on save.

### Reincarnate (`Reincarnate/`)
A second, permanent meta-progression currency ("Reincarnate Points") layered on top of the gold tree. From the death screen, the player can *reincarnate*: banking points and resetting the gold skill tree + wave counter to 1, while keeping every Reincarnate-tree upgrade forever.
- **`ReincarnateService`** (`Reincarnate/ReincarnateService.cs`) — scene **singleton** (`Instance`), implements `ISkillTreeService` against its own `SkillTreeSO` (pointed at `Config/Reincarnate.csv`), spending/persisting `SaveData.reincarnatePoints`/`purchasedReincarnateNodeIds` instead of gold. `PreviewPointsForReincarnate()` reads `WaveSpawner.Instance.CurrentWave` (points earned = highest wave reached this run). `Reincarnate()` banks those points, clears `SaveData.purchasedNodeIds` (so `SkillTreeService.Start()` naturally comes back empty next run), sets `RunState.StartingWave = 1`, and reloads the scene — it never touches its own purchased-node list, since the point tree is meant to persist.
- Starter Reincarnate nodes (`Config/Reincarnate.csv`): **Death Nova** (`DeathNova`/`DeathNovaSO`, see Player above), **Golden Goblin** (`GoldenGoblin`, see Enemies above), and **Grave Robber** (`GoldOnDeathCollector`, see Player above) — each a small independent branch, stacking 2-4 tiers deep.

## Conventions observed in this codebase

- **`Health` is the hub; dependencies point inward.** `Health` raises `OnDied` (once) and `OnDamaged(Damage)` and knows nothing about its listeners. Reactive behaviour (death animation, stopping movement/attacks, disabling colliders, dropping loot, scoring, damage numbers, UI, wave tracking) lives in separate components that **subscribe** to those events. When adding a reaction to damage or death, add a listener — don't reach into `Health`. Always **unsubscribe in `OnDestroy`**. The one exception is **`TryPreventDeath`**, an `event Func<bool>` checked right before a lethal hit latches — a handler that wants to *cancel* death (not just react to it), like `DeathNova`'s revive, returns `true` and calls `Health.Revive` itself. Its sibling **`TryBlockDamage`** (an `event Func<Damage, bool>`, checked before any health is lost) is the equivalent for negating a hit entirely, used by `DamageBlocker`. These two are the only hooks that can alter what `Health` does; everything else stays a normal reactive listener.
- **Death is signalled, not destruction.** Dead enemies become corpses (colliders disabled, death/cheer animation) and the GameObject persists. Detect death via `Health.OnDied` / `Health.IsDead`, **never** via `OnDestroy` or object-count.
- **Validate dependencies in `Start`.** Auto-wire sibling/serialized references in `OnValidate` (`GetComponent<…>()`, or `GetComponentInChildren<Animator>()` since Synty rigs keep the Animator on a child), then in `Start` null-check, `Debug.LogError`, and set an `anyError` flag; `Update`/handlers early-return `if (anyError)`. Follow this for any new component.
- **Scene singletons**: `Player.Instance`, `GameStats.Instance`, `WaveSpawner.Instance` — assigned in `Awake`, cleared in `OnDestroy`, and they reset naturally on scene reload. Reach other objects through these rather than `FindObjectOfType`.
- **Tunables go on a `ScriptableObject`** (`*SO`, `[CreateAssetMenu]` under `Scriptable Objects/…`). Components read shared config from an asset rather than per-instance fields. Tune behaviour by editing the asset, not the script.
- **Numbers that might plausibly become upgrade-modifiable are registered as `StatType` bases, not hardcoded literals** — even when the current value is 0/off (e.g. `GoldenGoblinChance` starts at 0 until a Reincarnate node raises it; `DeathNovaCharges` starts at 0 to mean "locked"). This mirrors the existing `SwordDamage`/`MoveSpeed` pattern: whichever system owns the value registers the base via `PlayerStats.SetBase` and reads it back via `GetValue`, so any future skill/Reincarnate node can layer a modifier on top without that system's code changing at all.
- **Animator integration by trigger name**: death/cheer/attack states are driven by string triggers (`Death`, `Cheer`, `Attack`) hashed in `Start`; timing-critical hits use either a tuned wind-up delay or a Unity animation event routed through `AnimationEvents`.
- **Persistence has two tiers**: cross-run progress (gold) goes through `SaveSystem`/`SaveData` to disk; within-session run state that must survive a scene reload (current wave) goes in a static like `RunState`. Restarting = `SceneManager.LoadScene` of the active scene, which resets scene singletons.
