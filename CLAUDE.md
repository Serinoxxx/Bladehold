# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Overview

Unity 6 game project, codenamed **Bladehold**. A 3D game built on Unity's Universal Render Pipeline (URP) with the new Input System and NavMesh-based AI. Unity Editor version is pinned in `ProjectSettings/ProjectVersion.txt` (currently `6000.3.10f1`) — open with exactly this version.

## Building, running, and testing

There is no command-line build script. Day-to-day work happens in the Unity Editor:

- **Open the project**: open the project folder in Unity Hub with the matching Editor version.
- **Play / run**: enter Play mode in the Editor. The only scene in the build list is `Assets/Scenes/SampleScene.unity` (see `ProjectSettings/EditorBuildSettings.asset`).
- **Build a player**: File > Build Profiles / Build Settings. URP renderers are pre-configured for two targets — `Assets/Settings/PC_Renderer.asset` and `Assets/Settings/Mobile_Renderer.asset`.
- **Tests**: `com.unity.test-framework` is installed. Run via the Editor's **Test Runner** window (Window > General > Test Runner). There are no test assemblies in `Assets/` yet — a new one needs its own `.asmdef` to be picked up.

The `.csproj` and `.slnx` files are **generated by Unity** for IDE integration (Rider / Visual Studio). Do not hand-edit them; edit code under `Assets/` and let Unity regenerate. C# compiles automatically on focus/save in the Editor.

## Code layout

First-party game code lives **only** in `Assets/Bladehold/Bladehold/`. Everything under `Assets/Third Party/` (Synty animations, Unity StarterAssets ThirdPersonController, Wingman editor tool) is vendored and should generally not be modified.

There are **no `.asmdef` files for game code**, so all first-party scripts compile into the default `Assembly-CSharp`. The only first-party-adjacent assembly definition is `Assets/Third Party/Wingman/Wingman.asmdef` (an editor productivity tool, not game logic).

## Architecture

The gameplay code is small and centers on enemy AI chasing the player.

- **`Player`** (`Player/Player.cs`) — a singleton (`Player.Instance`, set in `Awake`). Enemies locate the player through this static reference rather than scene lookups. There is currently no movement logic on it beyond the singleton wiring.

- **`AIMovementSO`** (`Enemies/AIMovementSO.cs`) — a `ScriptableObject` (`Scriptable Objects/AIMovementSO` in the Create menu) holding tunable AI parameters: pathfinding (`speed`, `updateInterval`), animation gait speeds (`walkSpeed`/`runSpeed`/`sprintSpeed`, used to derive gait thresholds) and an optional `leanCurve`. Movement/animation components read shared config from an asset (e.g. `Enemies/AIMovementSO.asset`) rather than per-instance fields. **Tune AI behavior by editing the SO asset**, not the scripts.

- **`AIMovement`** (`Enemies/AIMovement.cs`) — drives a `NavMeshAgent`. On `Start` it copies `movementSO.speed` onto the agent and caches `Player.Instance`. In `Update` it re-issues `agent.SetDestination(player.position)` only every `updateInterval` seconds (throttled pathfinding to limit cost).

- **`LocomotionAnimator`** (`Enemies/LocomotionAnimator.cs`) — a plain (non-`MonoBehaviour`) reusable class extracted from Synty's `SamplePlayerAnimationController`. It computes the goblin locomotion blend-tree parameters (`MoveSpeed`, `CurrentGait`, `IsStopped`/`IsStarting`, strafe/shuffle blend, `LeanValue`, …) from a per-frame `LocomotionInput` snapshot and writes them to an `Animator`. It is input-agnostic — fed by a `NavMeshAgent` for AI, but equally usable from player input/camera — and never moves or rotates the transform itself. Camera-only outputs (head/body look) are intentionally left at their defaults.

- **`AIAnimation`** (`Enemies/AIAnimation.cs`) — drives the goblin `Animator` for an AI character by mapping its `NavMeshAgent` motion onto a `LocomotionAnimator.LocomotionInput` each frame (velocity, path-derived move direction, forward; grounded, non-strafing). Gait thresholds and the optional lean curve come from `AIMovementSO`.

- **`Enemy`** (`Enemies/Enemy.cs`) — currently an empty stub.

Navigation depends on a baked NavMesh in the scene (`com.unity.ai.navigation`); AI agents will not move without it.

## Conventions observed in this codebase

- **Validate dependencies in `Start`**: each AI component auto-wires sibling components in `OnValidate` (e.g. `GetComponent<NavMeshAgent>()`), then in `Start` checks for missing references, `Debug.LogError`s, and sets an `anyError` flag. `Update` early-returns `if (anyError)`. Follow this pattern when adding components that depend on `[SerializeField]` references or sibling components.
- Inter-object references go through the `Player.Instance` singleton; prefer this over `FindObjectOfType`.
- New tunable gameplay values belong on a `ScriptableObject`, mirroring `AIMovementSO`.

## Key packages

URP 17.3 (`com.unity.render-pipelines.universal`), new Input System 1.18 (`com.unity.inputsystem`) with actions in `Assets/InputSystem_Actions.inputactions`, AI Navigation 2.0, Cinemachine 3.1, Timeline, and Visual Scripting. Full list in `Packages/manifest.json`.
